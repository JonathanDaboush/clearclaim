================================================================================
                        CLEARCLAIM — ENTITY RELATIONSHIP DIAGRAM
================================================================================


  ┌─────────────────────────────────────────────────────────────────────────┐
  │                              LEGEND                                    │
  │                                                                        │
  │   PK  = Primary Key          FK  = Foreign Key                         │
  │   UQ  = Unique Constraint    NN  = Not Null                            │
  │   ?   = Nullable             []  = JSON                                │
  │                                                                        │
  │   Data types:                                                          │
  │     UUID      = universally unique identifier                          │
  │     TEXT      = variable-length string                                 │
  │     BOOL      = true / false                                           │
  │     INT       = integer                                                │
  │     ENUM      = restricted set of values                               │
  │     TIMESTAMP = date + time (UTC)                                      │
  │     DATE      = date only                                              │
  │     DECIMAL   = exact numeric (for money)                              │
  │     JSON      = structured data (arrays or objects)                    │
  │                                                                        │
  │   Relationships:                                                       │
  │     ──────>>   one-to-many   (one row links to many rows)              │
  │     ── 1:1 ──  one-to-one                                             │
  │     many-to-many = linked through a join table                         │
  │                                                                        │
  │   All PKs are UUID. All timestamps are UTC.                            │
  └─────────────────────────────────────────────────────────────────────────┘

  7. AUDIT LOGS (append-only, cryptographically chained)
================================================================================

  ┌──────────────────────────────────────────────────────────────┐
  │                    AUDIT_LOGS                                │
  ├──────────────────────────────────────────────────────────────┤
  │  PK      id                      UUID                          │
  │  NN     event_type              TEXT                          │
  │  FK,NN  user_id                 UUID ──>> users.id            │
  │  FK,?   device_id               UUID ──>> devices.id          │
  │  FK,?   contract_id             UUID ──>> contracts.id        │
  │  FK,?   contract_version_id     UUID ──>> contract_versions.id│
  │  NN     timestamp               TIMESTAMP (UTC)               │
  │  ?      details                 TEXT                          │
  │  NN     prev_hash               TEXT (required — chain link)  │
  │  NN     hash                    TEXT                          │
  │  ?      snapshot_hash           TEXT                          │
  └──────────────────────────────────────────────────────────────┘

  Chain rule: hash = H(event_type + user_id + timestamp + details + prev_hash)
  Snapshot hashes anchored externally for periodic integrity verification.


================================================================================
  8. KEY MANAGEMENT
================================================================================

  ┌──────────────────────────────────────────┐
  │               KEYS                       │
  ├──────────────────────────────────────────┤
  │  PK     id            UUID                 │
  │  NN    type          ENUM                 │
  │  FK,?  parent_id     UUID ──>> keys.id    │  (nullable — self-referential)
  │  NN    created_at    TIMESTAMP            │
  │  ?     rotated_at    TIMESTAMP            │
  │  NN    status        ENUM                 │
  └──────────────────────────────────────────┘

  Key hierarchy:
    root (offline/HSM)
      └── system (HSM, signs audit entries)
            └── session (ephemeral, per device/document)

  Rotation: new key signed by previous key. Parent chain preserved.
